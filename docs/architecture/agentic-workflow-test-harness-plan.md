# Agentic Workflow Test Harness Plan

Date: 2026-05-22

## Purpose

Studio and HGO now have enough browser-local workflow surface that Codex and
future agents need a repeatable way to test the workbench without relying only
on human click-through notes. The harness should prove that synthetic source
material can move through Studio manuscript helpers, manuscript-library payloads,
HGO projection export, HGO validation, and preview-import payload shape while
producing a machine-readable report.

This is not a replacement for manual editorial review. It is a guardrail that
makes the source/tag/projection boundary easier for agents to verify before
anyone uses real manuscript material.

## What Codex Should Be Able To Test

- A synthetic Studio manuscript draft can be created without real text.
- The draft is recognized as synthetic smoke material.
- A local-only manuscript-library create payload can be derived from the draft.
- A local-only manual snapshot payload can be derived from the draft.
- HGO episode projection JSON can be generated from tagged synthetic manuscript
  data.
- The projection omits raw draft internals such as `editorJson`,
  `quoteReviews`, `structureRegions`, and inline `marks`.
- The HGO validator accepts the projection while warning about staged review,
  pull quotes, and unresolved citation states.
- A JSON report records steps, warnings, errors, routes, commit SHA, and safety
  confirmations.

## What Remains Manual For Now

- Authenticated Studio browser operation through Google OAuth / NextAuth.
- Real server snapshot creation from a browser session.
- Confirming browser replacement prompts after loading a snapshot.
- Visual verification that `/projection-preview/import` renders exactly as
  expected.
- Any smoke involving real manuscript material.

## Auth Strategy

There is no committed Playwright setup and no safe pre-authenticated Studio
browser storage state in the repo. The always-available harness is therefore
API/helper level and local-only.

The operator-assisted browser harness uses Playwright only when a private
storage-state file exists at:

```text
artifacts/auth/studio-storage-state.json
```

When that file is missing, `pnpm studio:hgo:browser-smoke` writes a `blocked`
report and exits without opening a browser or performing server writes.

The HGO-only browser harness does not need Studio auth. It runs:

```bash
pnpm hgo:projection:browser-smoke
```

The command uses synthetic HGO projection JSON, opens
`/projection-preview/import`, confirms validation warnings, confirms the shared
projection renderer mounts, checks known real-content markers are absent, and
writes a JSON report. If `HGO_BASE_URL` is not provided, it starts the `web`
app locally on an available test port and shuts it down after the run.

The HGO visual artifact harness also does not need Studio auth. It runs:

```bash
pnpm hgo:projection:visual-smoke
```

The command visits synthetic-only HGO projection preview surfaces, captures
full-page screenshots, writes a route-matrix JSON report, and performs no
server writes. It is for later human visual review of the renderer and staged
preview direction, not for validating real publishing.

Future browser runs should use one of these safe paths:

- Local test auth if Studio gains a deliberate non-production test-auth mode.
- A documented, private Playwright storage-state file that is generated by an
  operator and never committed.
- A production read-only smoke that stops before any server write and never
  tries to automate Google OAuth.

The harness must not hack around OAuth, scrape credentials, print secrets, or
create an alternative production auth path.

## Test Data Strategy

The harness uses `createSyntheticManuscriptSmokeDraft()` only. It may mutate
synthetic metadata locally to simulate a browser-local change, but it must not
read real manuscript files, public HGO content, staging content, inbox content,
or canonical content records.

Synthetic server payloads may be shaped locally for validation, but the default
agentic harness performs no network calls and no server writes.

The operator-assisted browser harness may create synthetic-only
`StudioManuscript` and manual `StudioManuscriptSnapshot` records when valid
auth state is supplied. The report must say that clearly. It must not use real
manuscript text or real HGO content.

## Report Strategy

The command writes:

```text
artifacts/agentic-smoke/studio-hgo-smoke-report.json
```

The browser command writes:

```text
artifacts/agentic-browser-smoke/studio-hgo-browser-smoke-report.json
```

The HGO no-auth browser command writes:

```text
artifacts/agentic-browser-smoke/hgo-projection-browser-smoke-report.json
```

The HGO visual artifact command writes:

```text
artifacts/agentic-browser-smoke/hgo-projection-visual-smoke-report.json
```

Latest screenshots are written to:

```text
artifacts/playwright/hgo-projection-visual-smoke/
```

The report includes:

- `startedAt` / `finishedAt`
- `status`
- `environment`
- `steps`
- `screenshots`
- `errors`
- `warnings`
- `routesTested`
- `commitSha`
- projection and validation summary
- safety confirmations for synthetic-only data, no server writes, no publish
  action, no autosave, and no Yjs/collaboration

The visual report additionally includes `routes[]` entries with path, status,
screenshot, title, heading, and notes for each captured preview surface.

The report directory is generated output and is ignored by git.

Browser screenshots, when available, live under `artifacts/playwright/` and are
also ignored.

## Future Path

- Add a Playwright browser smoke once the repo has a safe auth strategy.
- Add optional screenshots/traces for local authenticated browser runs.
- Extend visual smoke coverage to mobile/tablet screenshots once the desktop
  artifact pass stays stable.
- Add a production read-only smoke that checks page health and static route
  availability without server writes.
- Add CI-safe smoke that runs only helper-level checks and never depends on
  secrets, OAuth, Cloud SQL, or Cloud Run.
- Extend the HGO no-auth browser smoke into CI once browser installation and
  dev-server startup are stable in the CI runner.
